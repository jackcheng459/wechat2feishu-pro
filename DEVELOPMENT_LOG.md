# 📔 WeChat2Feishu 开发沉淀日志 (Development Log)

这是一份记录了项目从 0 到 1 开发过程中核心技术攻关、架构抉择及“填坑”经验的珍贵文档。它不仅是代码的补充，更是我们共同解决复杂问题的记忆。

---

## 📅 项目时间轴：2026-03-09

### 1. 🖼️ 图片防盗链的“终极绝杀”
**背景**：微信公众号图片具有极高的防盗链强度，常规抓取 URL 后再上传会导致 403 错误或生成 `default.png`。
**技术突破**：
- **方案**：利用 `Playwright` 监听网络响应 (`page.on("response", ...)`）。
- **经验**：在浏览器渲染页面的第一现场，直接拦截图片的二进制原始数据流。
- **沉淀**：通过 `base64` 编码暂存在内存中，彻底避开了“二次请求”被封禁的可能。

### 2. 🏗️ “替身修补”战术 (Two-Step Patch Strategy)
**背景**：飞书 `import_tasks` API 虽然排版完美，但对图片链接校验极严，且导入后会重新生成 token，导致映射丢失。
**逻辑演进**：
- **旧路**：直接在 Markdown 里塞原图 URL（失败：防盗链拦截）。
- **中路**：先传图片拿 token 塞进 Markdown（失败：飞书拒绝非 docx 类型的 token）。
- **绝杀路**：
    1. **第一步（替身）**：用 1x1 像素透明图作为占位符生成文档，骗过导入校验并锁定排版位置。
    2. **第二步（修补）**：文档生成后，读取 Block 列表，按抓取顺序通过 `Docx Block Patch` 接口将真图“硬塞”进去。
- **经验**：该方案完美平衡了“飞书官方高保真排版”与“本地图片注入”的矛盾。

### 3. 🛡️ 攻克飞书 Wiki 的“隐形迷宫”
**背景**：文档转存至 Wiki 知识库时，常出现“内容全空”或“图片不可见”的灵异现象。
**填坑经验**：
- **内容丢失**：不要用 `origin` 模式在 Wiki 节点新建，而要先在云空间创建完整文档，再使用专用接口 `move_docs_to_wiki` 进行“物理搬迁”。
- **图片权限**：Wiki 文档与个人空间 Docx 存在权限隔离。在 Patch 图片时，**必须**在 `extra` 参数中注入 `drive_route_token`（值为文档底层的 `document_id`），否则图片无法渲染。

### 4. 🧹 Markdown 的“手术刀级”清洗
**背景**：微信 HTML 转换出的 Markdown 存在严重的排版粘连问题。
**清洗法则**：
- **内容隔离**：检测到 `![图片](...)### 标题` 紧连时，强制插入双换行，确保标题顶格识别。
- **JSON 净化**：微信常将配置代码压缩为单行。我们建立了“大括号换行恢复”逻辑，结合正则美化，让 scopes 权限列表在飞书中清晰可见。
- **空格规整**：统一清理 `\xa0` 等不可见字符，防止飞书解析器罢工。

### 5. 📂 本地备份的“中庸之道”
**抉择历程**：
- **曾尝试**：全 Base64 内联 MD。结论：源码全是乱码，人类无法编辑。
- **最终方案**：回归“README.md + images/”结构。
- **理由**：既保证了本地知识库的纯净，又实现了 100% 的离线可见性，且方便用户在本地进行二次编辑。

---

## 💡 核心箴言 (Lessons Learned)
1. **不要试图对抗飞书的排版引擎**：交给 `import_tasks` 去做，我们只负责喂给它标准的数据。
2. **Wiki 里的文档不只是文档**：它是节点、实体和权限的集合体，移动比新建更稳。
3. **拦截永远比事后弥补有效**：在浏览器渲染那一刻拿到的数据是最真实的。
