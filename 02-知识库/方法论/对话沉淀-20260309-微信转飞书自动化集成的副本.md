# 对话沉淀 — 2026-03-09 — 微信公众号至飞书高保真自动化转存项目实战

---

## 【正文】

### 一、决策记录
1. **采用“替身修补”战术 (Two-Step Patch)**：由于飞书 `import_tasks` API 对外链图片的强力拦截和对 Token 的严格校验，决策先使用 1x1 透明图占位生成文档排版，再通过 `Docx Block Patch` 接口回填高保真原图。
2. **Wiki 挂载逻辑调整**：由于传统的“创建节点”接口在挂载现有文档时极易产生空文档，决策改用专门的 `move_docs_to_wiki` 接口，实现文档内容的物理搬迁而非结构复制。
3. **本地备份结构回归**：经过 Base64 全内联模式的尝试，发现其严重破坏了 MD 源码的可读性，最终决策回归“README.md + images/”的经典解耦结构，平衡了离线可见性与编辑便利性。
4. **自动化静默增强**：为了解决“无人值守”时的浏览器弹窗问题，决策引入 `USE_TENANT_MODE`（应用令牌模式），作为用户授权失效时的应急兜底。

### 二、认知收获
1. **认识到 Markdown 的局限与编辑器的能力边界**：理解了 Markdown 本质上是纯文本，其“图文并排”显示完全依赖于预览引擎的渲染，而非文件本身的物理合并。
2. **理解了飞书 Wiki 的权限路由机制**：深刻理解了 `drive_route_token` 的作用——在知识库这种高度隔离的权限环境下，图片资源必须持有底层文档的“入场券”方可渲染。
3. **认识到拦截优于重抓**：在对抗强力防盗链（如微信）时，与其研究如何模拟请求下载图片，不如在浏览器渲染的第一现场直接截获已加载的二进制流。

### 三、产出清单
1. **代码库**：[jackcheng459/wechat2feishu](https://github.com/jackcheng459/wechat2feishu)（已开源）。
2. **核心模块**：
    - `feishu.py`: 支持两步法注入图片、Wiki 完美挂载、比例自适应。
    - `processor.py`: “手术刀级”Markdown 清洗器，修复粘连标题与截断代码块。
    - `main.py`: 集成了飞书云端转存与本地文件夹自动备份的双轨逻辑。
3. **文档**：
    - `DEVELOPMENT_LOG.md`: 深度记录了四大技术攻关点的实战记忆。
    - `README.md`: 标准的开源项目说明书。
4. **OpenClaw 技能**：`wechat2feishu.md` 技能定义文件，已完成路径绝对化与调用黑盒化。

### 四、踩坑记录
1. **0字节图片坑**：微信图片 URL 参数变动导致爬虫缓存匹配失败，兜底下载又触发防盗链，导致上传了大量空文件。*解决：统一基础 URL 模糊匹配。*
2. **Invalid Param (1770001)**：飞书 Docx API 在不同上下文下对 `token` 和 `file_token` 的 Key 要求不一致。*解决：通过多轮调试确认为 file_token 为 Patch 接口的真命天子。*
3. **Wiki 空白文档坑**：使用 `origin` 模式挂载导致飞书新建了空壳文档。*解决：改用 `move_docs_to_wiki` 接口并增加索引同步延迟。*

### 五、反思
*   **不与原生引擎对抗**：处理复杂排版时，最好的策略是利用官方最强大的入口（如 `import_tasks`），将我们的智慧集中在“数据清洗”和“后期修补”上，而不是试图自己重写一套完美的排版引擎。
